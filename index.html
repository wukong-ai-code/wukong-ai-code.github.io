核心架构

五大支柱：打造真正有用的 AI 智能体

大多数 AI 工具是被动响应的——你问，它答，然后忘记一切。 Hermes Agent 围绕五个核心支柱构建，让 AI 从"工具"进化为"伙伴"。

🧠

Memory · 记忆

「记忆是把有用助手和华丽搜索引擎区分开的东西。」

分层存储（向量语义搜索 + 键值事实存储），配合 RAG 动态检索。 不是把所有历史塞进上下文——只拉取当前对话真正需要的那条记忆。 跨会话持久化，你不需要重复解释自己。

📚

Skills · 技能

「一个能描述怎么发邮件却真发不出去的 AI，有严重缺陷。」

可调用、可组合的离散功能模块。支持 Agentskills.io 标准，技能可分享、社区共建。 Agent 自己写技能、优化技能、在需要时自动加载——过程记忆，不是硬编码。

🎭

Soul · 灵魂

「不一致的行为会侵蚀信任。灵魂文件是每个会话的锚。」

一个结构化配置文件（SOUL.md），定义 Agent 的核心人格、语气、行为边界。 每次会话开始时加载，作为不变的角色基石——即使 Agent 在适应不同任务时也不会漂移。

⏰

Crons · 定时任务

「工具响应命令，助手预判需求。」

自然语言描述的定时任务引擎。可以按 cron 表达式或简单短语（"每天早上 8 点"）调度。 日报摘要、数据监控、收件箱分拣——Agent 在你开口之前就开始工作了。

🔄

Self-Improvement · 自我进化

「一个静态的 Agent 会随时间贬值，即使世界在变。」

闭环学习：用户层适应 → 技能层优化 → 记忆层精炼 → 灵魂层校准。 Agent 会定期反思对话、识别模式、提出配置更新建议（有人工审核门控）。 这是 Hermes 与所有其他 Agent 的根本区别。

技术架构

一图看懂 Hermes Agent 内核

模型无关、平台无关、后端无关。一切都是可插拔的。

CLI/TUI Telegram Discord Slack WhatsApp 微信 飞书 Cron API/Batch 入口层 — 适配器模式，核心逻辑不感知平台
↓
AIAgent 核心循环 build_system_prompt → call model → dispatch tools → repeat
↓
Tools Registry Skills Loader Memory Manager Model Providers Exec Backends 模块层 — 自注册、可组合、渐进式加载（Level 0 → 1 → 2）
↓
SQLite SessionDB (WAL + FTS5) Markdown Mem/Skills Docker/SSH/Modal 持久层 — 一切数据都在本地。你的 Agent，你的数据。

设计原则：

① 平台无关核心 — 适配器翻译平台事件，核心不关心消息从哪来

② Prompt 缓存友好 — System Prompt 在会话开始时冻结，整个会话期间不变

③ 渐进式披露 — 先加载 Skill 概要（Level 0），需要时才加载全文（Level 1-2）

④ Agent 自己管理学习产物 — 不是人写 Skill，是 Agent 写 Skill

⑤ Profile 隔离 — 每个实例独立 HERMES_HOME，多身份互不干扰

经验干货

来自社区和一线用户的实战心得

💡

选对模型上下文窗口

大多数首次启动问题来自选择了一个上下文窗口太小的模型。Hermes 的 System Prompt + Memory + Skills 需要一定空间。建议至少 128K 上下文。

💡

先从 Docker 后端开始

不熟悉安全配置？用 docker 后端跑 Hermes。一次配置，隔离一切。等熟悉了再切 local/SSH。

💡

善用 Skill 的渐进加载

Skills 太多会让 System Prompt 膨胀。遵循 Level 0 → 1 → 2 设计：只描述功能，需要时才加载详细步骤和参考文件。节省 40-60% token 开销。

💡

Cron + no_agent 做纯监控

磁盘/内存/SSL 监控不需要 LLM 推理。用 no_agent=true + shell 脚本——零 token 消耗，毫秒级响应。只有非零退出码时才通知你。

💡

记忆要「少而精」

不要什么都存进 Memory。最好的记忆是防止你下次需要纠正 Agent 的那些事实。用户偏好 > 环境配置 > 过程知识。任务进度、PR 编号不要存。

💡

网关凭证权限最小化

Telegram Bot Token、Discord Bot Token 等，只给必要权限。不要把 Bot 加到所有频道——用 per-user authorization 白名单模式。

常见问题

你可能会问

Hermes Agent 和 ChatGPT / Claude 有什么区别？

ChatGPT 是聊天机器人，你每次对话它都重新开始。Hermes 是持久化自主智能体——它有自己的文件系统、终端、记忆系统、定时任务、技能库。它知道你上周做了什么、记得你的偏好、能在你睡觉时自动运行任务。把它想成一个住在你服务器上的数字员工，而不是一个浏览器里的聊天窗口。

「自我进化」是真的吗？怎么做到的？

是的。当 Hermes 解决了一个复杂问题，它会自动把解决方案写成 Skill 文档（skill_manage）。下次遇到类似问题，自动加载对应 Skill。同时它会不断更新 MEMORY.md / USER.md，优化记忆检索质量，并在适当时机反思最近的对话、找出可改进的模式。这不是「模型在变强」——而是围绕模型的上下文系统在变强。效果是：你用得越久，它越「懂」你，做起事来越高效。

我的数据安全吗？

所有数据都在你自己的机器上。不经过任何第三方服务器。对话历史存在本地 SQLite，记忆和技能是本地 Markdown 文件。如果你想更隔离，用 Docker 后端（容器加固 + 命名空间隔离）。还有命令审批模式（approval mode），对危险操作需要你的确认。简单说：你是数据的唯一拥有者。

需要什么配置才能跑？

最低：一台能跑 Linux/macOS/WSL2 的机器（树莓派都行）+ 任意 LLM API key。推荐：2GB+ RAM、128K 上下文窗口的模型。云端部署推荐 2 vCPU + 4GB RAM 的轻量云服务器（腾讯云轻量、AWS Lightsail 等），月费不到 $5。Serverless 模式（Daytona/Modal）下闲置成本几乎为零。

可以用国产模型吗？

完全可以。Hermes 支持 Nous Portal（300+ 模型，包括国产）、智谱 GLM、Kimi（月之暗面）、MiniMax、小米 MiMo、Z.AI 等国产模型提供商。用 hermes model 一键切换。关键是选一个上下文窗口够大（128K+）且工具调用能力好的模型。

能用在生产环境吗？

取决于你的容错需求。Hermes 是 MIT 开源的，企业可以自由使用和修改。社区已有多个生产级案例：B2B 外贸 AI 助手（14 个业务 Skills）、团队 Telegram 助手、自动化运维 watchdog、内容创作流水线等。Petronella Technology Group 等公司提供托管部署服务。建议从低风险场景（内部工具、自动化）开始，验证稳定性后再扩展到关键业务。

Daily Workflow

波布画像：把 Hermes 当成日常工作台

波布不是偶尔打开 AI 问一句的人。他把 Hermes 放进每天的工作流：查资料、比模型、做页面、迁移技能、发布到 GitHub，也让它记住自己的节奏，提醒、复盘、沉淀知识。

日常入口

微信就是入口。想到一个问题，直接丢给 Hermes：AI 行业动态、模型选择、配置排错、页面修改、发布上线。它像一个随手可叫的工作伙伴，不需要正式启动仪式。

判断方式

波布先看事实，再看信号，最后要结论。喜欢表格、分类和信号解读；讨厌空泛判断。一个答案如果不能验证、不能复用、不能继续推进，就还不够好。

高频任务

调研 AI 趋势，比较模型和供应商；生成 landing page、PNG、结构化报告；排查 Hermes 配置、Provider、Gateway；迁移 OpenClaw 技能、记忆和日常流程；把结果推到 GitHub 分享。

生活节奏

Hermes 不只处理工作，也承接日常：工作、学习、饮食、健身、作息提醒和周期复盘。它记住偏好，把重复事项变成提醒，把重复方法沉淀成技能。
